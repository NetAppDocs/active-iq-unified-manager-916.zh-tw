= 響應由 QoS 策略群組限制引起的動態效能事件
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
您可以使用 Unified Manager 調查服務品質 (QoS) 策略群組限制工作負載吞吐量 (MB/s) 所導致的效能事件。限制增加了策略群組中磁碟區工作負載的回應時間（延遲）。您可以使用事件資訊來確定是否需要對策略群組施加新的限制以停止限制。

.開始之前
* 您必須具有操作員、應用程式管理員或儲存管理員角色。
* 必須有新的、公認的或過時的績效事件。


.步驟
. 顯示*活動詳情*頁面以查看活動的資訊。
. 閱讀*描述*，其中顯示受限制影響的工作負載的名稱。
+
[NOTE]
====
描述可以顯示受害者和霸凌者的相同工作負載，因為限制使工作負載成為其自身的受害者。

====
. 使用文字編輯器等應用程式記錄磁碟區的名稱。
+
您可以稍後搜尋卷名來找到它。

. 在「工作負載延遲」和「工作負載利用率」圖表中，選擇「Bully Workloads」。
. 將遊標停留在圖表上，即可查看影響策略群組的主要使用者定義工作負載。
+
清單頂部的工作負載偏差最大，導致發生節流。活動是每個工作負載所使用的策略群組限制的百分比。

. 在「建議的操作」區域中，按一下排名靠前的工作負載的「分析工作負載」按鈕。
. 在工作負載分析頁面中，設定延遲圖表查看所有叢集元件，設定吞吐量圖表查看細分。
+
細分圖表顯示在延遲圖表和 IOPS 圖表下方。

. 比較*延遲*圖表中的 QoS 限制，以查看事件發生時多少限制量影響了延遲。
+
QoS 策略群組的最大吞吐量為每秒 1,000 個作業 (op/sec)，其中的工作負載總和不能超過該吞吐量。事件發生時，策略群組中的工作負載的合併吞吐量超過 1,200 操作/秒，這導致策略群組將其活動限制回 1,000 操作/秒。

. 將*讀取/寫入延遲*值與*讀取/寫入/其他*值進行比較。
+
兩個圖表都顯示讀取請求數量較多且延遲較高，但寫入請求的數量和延遲量較低。這些值可協助您確定是否存在大量吞吐量或大量操作增加了延遲。在決定對吞吐量或操作設定策略群組限制時，您可以使用這些值。

. 使用ONTAP系統管理員將策略組的目前限制增加到 1,300 op/sec。
. 一天后，返回 Unified Manager 並在 *工作負載分析* 頁面中輸入您在步驟 3 中記錄的工作負載。
. 選擇吞吐量細分圖表。
+
顯示讀取/寫入/其他圖表。

. 在頁面頂部，將遊標指向更改事件圖示 (image:../media/opm_change_icon.gif["更改事件圖標"] ) 來更改策略組限制。
. 將*讀取/寫入/其他*圖表與*延遲*圖表進行比較。
+
讀寫請求相同，但節流已停止，延遲也減少了。
